# CLI reference

pimm has two launch layers:

- **`pimm launch` / `pimm submit`** compose and run local or Slurm jobs from YAML
  launch settings. `pimm launch` runs locally (or inside an existing allocation);
  `pimm submit` submits a managed Slurm job through submitit.
- **`scripts/train.sh` / `scripts/test.sh`** are the direct execution wrappers
  used by the launchers and by older manual workflows.

For new runs, prefer the launchers when Slurm, containers, site paths, resource
overrides, or chained jobs are involved. Use the scripts directly for simple
local development or exact control over an existing shell workflow.

## Entry points

The installed console command is `pimm` (defined in `pyproject.toml` as
`pimm = "pimm.cli.main:main"`), which dispatches a small command namespace:

```{list-table}
:header-rows: 1
:widths: 22 78

* - Command
  - Purpose
* - `pimm launch`
  - Run training locally or inside an existing allocation (**LOCAL only**).
* - `pimm submit`
  - Submit training to Slurm through submitit.
* - `pimm export`
  - Export a checkpoint to a portable pretrained artifact.
```

These are equivalent:

```bash
pimm launch ...
python -m pimm.cli launch ...
```

If `pimm` is not on `PATH`, install the checkout into the active environment with
`pip install -e .`. The console script must exist on every login/submit host
where you type `pimm`.

:::{important}
There are **no** `dry-run` / `run` / `submit` subcommands under `pimm launch`.
Dry-run is a boolean flag (`--dry-run`) on both `pimm launch` and `pimm submit`.
`pimm launch` **errors if the resolved site is non-local** â€” use `pimm submit`
for Slurm.
:::

## How the launchers resolve settings

`pimm launch` / `pimm submit` load settings in this order (later wins):

1. `launch/defaults.yaml`
2. `launch/sites/<site>.yaml`
3. an optional run recipe passed with `--recipe PATH`
4. CLI overrides (Tyro flags) and post-`--` training overrides

Python files under `configs/` remain the source of truth for model/dataset/
trainer behavior. Launch YAML should describe *execution*: site paths, Slurm
resources, container runtime, checkpoint weights, resume choices, run naming, env
vars, and explicit training overrides.

### Tyro flags are nested and dotted

CLI flags are generated by Tyro from the dataclasses in `pimm/launch/schema.py`,
so they are **nested**: `--train.config`, `--resources.nproc-per-node`,
`--run.name`, `--slurm.account`. There are no flat `--config`, `--option`, or
`--set` flags.

```{list-table}
:header-rows: 1
:widths: 38 32 30

* - Setting
  - Flag
  - Notes
* - training config
  - `--train.config`
  - path under `configs/`, `.py` optional
* - checkpoint weight
  - `--train.weight`
  - passed through as `weight=`
* - resume
  - `--train.resume`
  - resume from latest complete checkpoint
* - skip code snapshot
  - `--train.no-code-copy`
  - run from live repo (dev mode)
* - GPUs per node
  - `--resources.nproc-per-node`
  - torchrun processes per node
* - nodes
  - `--resources.nnodes`
  -
* - walltime
  - `--resources.time`
  - `HH:MM:SS`
* - run name
  - `--run.name`
  - default: auto-generated `<config>-<timestamp>`
* - W&B display name / project
  - `--run.wandb-name` / `--run.wandb-project`
  -
* - Slurm account
  - `--slurm.account`
  - submit only
* - Slurm partition / QOS
  - `--slurm.partition` / `--slurm.qos`
  - submit only
* - run recipe
  - `--recipe PATH`
  - `launch/runs/*.yaml`
* - requeue attempts
  - `--chain.jobs N`
  - submit only (submitit requeue)
* - interactive salloc
  - `--interactive`
  - submit only
* - write rendered output
  - `--output PATH`
  - the launch script / submitit manifest
* - dry run
  - `--dry-run`
  - render without executing
```

### Config-path normalization

`--train.config` is normalized relative to `configs/` and without the `.py`
suffix. These are equivalent:

```bash
--train.config panda/pretrain/pretrain-sonata-v1m1-pilarnet-smallmask
--train.config configs/panda/pretrain/pretrain-sonata-v1m1-pilarnet-smallmask.py
--train.config /abs/path/configs/panda/pretrain/pretrain-sonata-v1m1-pilarnet-smallmask.py
```

After normalization `scripts/train.sh` splits the path into:

- config group: `panda/pretrain`
- config name: `pretrain-sonata-v1m1-pilarnet-smallmask`
- config file: `configs/panda/pretrain/pretrain-sonata-v1m1-pilarnet-smallmask.py`
- experiment directory: `exp/panda/pretrain/<experiment-name>/`

### Training overrides go after `--`

Config overrides are **bare `KEY=VALUE` tokens** passed after a `--` separator;
they become the training `--options` entries (parsed by
`parse_training_overrides` in `pimm/launch/config.py`, values via
`yaml.safe_load`):

```bash
pimm launch \
  --train.config panda/pretrain/pretrain-sonata-v1m1-pilarnet-smallmask \
  -- epoch=10 data.train.max_len=1000 optimizer.lr=3e-5
```

:::{warning}
Tokens after `--` must be `KEY=VALUE`; **any token starting with `--` is
rejected**. So `-- --options epoch=10` is INVALID for the launchers. The
`-- --options ...` form is only for direct `scripts/train.sh` invocations.
:::

## `pimm launch` â€” local / in-allocation

The default site is `local`:

```bash
pimm launch \
  --train.config panda/pretrain/pretrain-sonata-v1m1-pilarnet-smallmask \
  --resources.nproc-per-node 1 \
  --run.name local-smoke
```

`scripts/train.sh` uses `torchrun`, so a local multi-GPU run needs no Slurm:

```bash
pimm launch \
  --train.config panda/pretrain/pretrain-sonata-v1m1-pilarnet-smallmask \
  --resources.nproc-per-node 4
```

The rendered command calls:

```bash
sh ./scripts/train.sh -m <nodes> -g <gpus-per-node> -c <config> -n <run-name>
```

Use `--dry-run` to print that exact script without running it. `pimm launch` can
also run inside a user-authored Slurm allocation:

```bash
srun pimm launch \
  --resources.nnodes "$SLURM_NNODES" \
  --resources.nproc-per-node "$SLURM_GPUS_ON_NODE" \
  --train.config panda/pretrain/pretrain-sonata-v1m1-pilarnet-smallmask
```

## `pimm submit` â€” managed Slurm

```bash
pimm submit --site s3df \
  --train.config panda/pretrain/pretrain-sonata-v1m1-pilarnet-smallmask
```

Always dry-run before changing resources, accounts, sites, or recipes:

```bash
pimm submit --site s3df \
  --train.config panda/pretrain/pretrain-sonata-v1m1-pilarnet-smallmask \
  --resources.nnodes 2 \
  --resources.nproc-per-node 4 \
  --slurm.account neutrino:ml-dev \
  --slurm.partition ampere \
  --dry-run
```

The `--dry-run` output is the authoritative submitit manifest â€” a single YAML
document describing the job and its pre-rendered requeue attempts. Check the
rendered resource parameters, account, partition, and GPU request before
submitting. Site profiles (`launch/sites/<site>.yaml`), interactive `salloc`
runs, and requeue chaining are covered in {doc}`../hpc/index`.

## `pimm export` â€” portable artifacts

Export a training checkpoint to a portable, self-contained pretrained directory for
fine-tuning or Hugging Face upload:

```bash
pimm export --run-dir exp/panda/pretrain/my_run last ./artifacts/my-model
```

The two positional arguments are the checkpoint and the output directory; with
`--run-dir`, the checkpoint name (e.g. `last`, `model_best`) resolves under
`<run-dir>/model`, and the config is inferred from `<run-dir>/config.py` /
`resolved_config.json`. It accepts split checkpoint directories (`model/last`)
and legacy `.pth` files.

```{list-table}
:header-rows: 1
:widths: 32 68

* - Flag
  - Meaning
* - `--run-dir PATH`
  - experiment directory with `config.py` and `model/`; lets the checkpoint
    arg be a name like `last`
* - `--config PATH`
  - config to use when it cannot be inferred from the run dir
* - `--model-card PATH`
  - a `README.md` text file to include as the model card
* - `--push-to-hub REPO_ID`
  - upload to the Hugging Face Hub after export
* - `--public`
  - create a public Hub repo (default: private)
* - `--token TOKEN`
  - Hugging Face token for upload
* - `--device DEVICE`
  - device used while consolidating checkpoint tensors (default `cpu`)
* - `--no-safe-serialization`
  - write `model.bin` (torch pickle) instead of `model.safetensors`
* - `--dry-run`
  - print resolved paths and exit without writing
```

See {doc}`../checkpoints/index` for the exported artifact layout and the Hugging
Face workflow.

## Direct training â€” `scripts/train.sh`

```bash
sh scripts/train.sh -c <config-path> [options] [-- --options key=value ...]
```

The getopts string is `"p:c:n:w:g:m:r:a:Ch"` â€” there is **no** `--list` long
option; browse the `configs/` tree to find configs.

```{list-table}
:header-rows: 1
:widths: 16 84

* - Flag
  - Meaning
* - `-c CONFIG`
  - config path under `configs/`, with or without `.py`
* - `-n NAME`
  - experiment name (default: `<config-name>-<timestamp>`)
* - `-g GPUS`
  - GPUs per machine; defaults to all visible CUDA devices
* - `-m MACHINES`
  - number of machines; defaults to `1`
* - `-w WEIGHT`
  - checkpoint path passed as `weight=`
* - `-r true`
  - resume from the latest complete checkpoint
* - `-a NAME`
  - W&B run name override
* - `-p PYTHON`
  - Python interpreter
* - `-C`
  - dev mode; skip the code snapshot and run from repository source
```

```bash
sh scripts/train.sh -g 4 \
  -c panda/semseg/semseg-pt-v3m2-pilarnet-ft-5cls-fft \
  -n semseg-smoke \
  -- --options epoch=10 data.train.max_len=1000
```

`train.sh` loads `.env` from the repo root if present, copies `scripts`, `tools`,
and `pimm` into `exp/<config-group>/<name>/code/`, and runs from that snapshot for
reproducibility (use `-C` for live edits). If `MODEL_DIR` is set, physical
checkpoints go under `MODEL_DIR/.../model` and are symlinked back.

:::{warning}
Resume is strict: the experiment directory must already exist. The script uses
the saved `exp/.../<name>/config.py` and selects the newest complete checkpoint
among `last`, `last.prev`, `model_last.pth`. If none is complete, it exits rather
than silently restarting.
:::

## Direct testing â€” `scripts/test.sh`

```bash
sh scripts/test.sh -c <config-path> -n <experiment-name> [-w model_best]
```

```{list-table}
:header-rows: 1
:widths: 16 84

* - Flag
  - Meaning
* - `-c CONFIG`
  - config path under `configs/`, with or without `.py`
* - `-n NAME`
  - experiment name (default: `debug`)
* - `-w WEIGHT`
  - checkpoint stem under `exp/.../model` (default: `model_best`)
* - `-g GPUS`
  - parsed and printed, but testing is not wrapped in `torchrun`
* - `-m MACHINES`
  - parsed and printed for consistency with training
* - `-p PYTHON`
  - Python interpreter
```

It runs `pimm/test.py` with `PYTHONPATH` pointed at the experiment's saved code
snapshot and `--options save_path=<exp-dir> weight=<exp-dir>/model/<weight>.pth`,
so a test evaluates the code associated with the experiment, not the current
working tree.
